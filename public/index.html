What is a Stone
A "stone" is a cryptographically signed peice of data that is used as an object of value in centralized transactional sytems. It can be used to represent or store things of value.

It is designed to be real, portable and easy to transfer through everyday messaging systems such as email, sms, whatsapp etc.

Some examples of assets and valuables that can be represented or stored in a stone includes:

  • Digital money
  • Coupons
  • Vouchers
  • Tickets
  • Identity Information
  • Commodities
  • Copyrights
  • Patents
  • Certificates
  • Binary Files

A stone is made up of 5 parts known as blocks. They are meta, ownership, attributes, embeds and signatures blocks. All blocks are signed using JSON Web Signature standards with RSA-SHA256 algorithm.

What is a Host
A host is an organization responsible for providing an address service, ownership ledger and a public key directory for assets.

The host makes use of the ownership block to describe and determine ownership of a given stone. As the only block accessible to the host, it can modify it, sign it and return a newly updated Stone.

Issuers of assets that do not require a central body to manage ownership or prefers to manage ownership themselves can ignore the ownership block and a host.

The host must provide endpoints to receive and respond to commands from clients for basic operations like asset type creation, address creation, ownership change and verification, global stone locking, stone verification and public key retrieval.

Why Stone
As the world becomes more and more digital, the need for efficient systems to allow for quick, easy and safe transfer of digital assets is increasing in demand. This has lead to the development of truly innovative technologies such as Bitcoin, Stellar, Ripple, Nxt and many more.

Many of these new technologies strive to be decentralized and trust-less at expense of cheap and easy scalability enjoyed by centralized systems. The difficulty in scaling such systems to meet the needs of millions of people could discourage users from transferring assets digitally.

Stone is designed to be used by centralised systems in need of a way to describe simple or complex assets without having control over the issuance of these assets. Centralized transaction services using Stone will only offer an optional address system, public key store, transfer & verification services.

We hope that people interested in developing cryptographically secure, highly scalable, centralized digital asset systems do so without needing to use complex technologies to achieve their objectives.
Address
An address is an important feature of a host that allows stones to be owned. A stone whose ownership information refers to an address is said to be owned by the address.

Addresses are protected by RSA key pair, therefore all transaction relating to the address must be signed and authorized by using the address's private key to sign transactions.
Issuer
An issuer is anyone who has registered an asset with the host and has access to the RSA key pair associated with the asset.

Asset issuers are able to create instances of their assets outside the host. Users can verify the authencity of an asset by pulling the asset public key from the host.

Issuers are responsible for signing the meta, attributes and embeds blocks.
Blocks
A stone is represented in JSON format. It is made up of five properties know as "blocks". The names of these blocks are meta, ownership, attributes, embeds and signatures.
Structure of a stone object
			
{
   "meta":       	{},
   "ownership":  	{},
   "attributes": 	{},
   "embeds":	 	{},
   "signatures": 	{}
}
Meta: The meta block is where the header information about the stone and it's issuer is stored. This block must be signed by the issuer during creation.

Ownership: An optional block for describing ownership. This block is used by the host to determine and describe ownership. It must be signed by the host.

Attributes: This is the only place an issuer can store arbitrary data about a stone instance. It must be signed by the issuer.

Embeds: To support complex stones, this block allows the issuer to include one or more stones inside of a stone. This is suitable for assets that also come with additional assets. This block must be signed by the issuer.

Signatures: This is where the JWS signatures of the other blocks is stored. It is also the block that gets base64 encoded to make up the final stone token.

Each block accepts an object containing expected properties. For a stone to be considered valid, all blocks must pass validation checks. The signatures block is excluded from the signing and verification processes.

Structure of a signatures block
			
{
   "meta":       	"xxxxxxxx",
   "ownership":  	"xxxxxxxx",
   "attributes": 	"xxxxxxxx",
   "embeds":	 	"xxxxxxxx"
}
Meta Block
This is the header of a stone. It contains information about the stone instance and it's issuer.
Structure of a meta block
			
{
   "id":			"jh88hg3sxn43isjhdg0jsdjie03cu3",
   "type":  		"some-coupon",
   "created_at": 	1458623859
}
id (string): This is the unique id of the stone. It must be a 40 characters long. Preferrable a SHA1 hash of a UUID. This is used by the host and the issuer to recognize a stone instance.

type (string): This is the unique name of the asset. The asset must be known to the host to allow for proper verification.

created_at (number): A unix timestamp indicating when a stone instance was created.

Ownership Block
This block contains information about the ownership of a stone instance. The host uses this block to store and determine ownership of an asset by associating the stone with an address hosted with it.

Only the address owner who has access to the address's private key can authorize transactions against a stone belonging to it's address.

This block is optional. Only required when a host is needed to manage asset ownership. Otherwise, issuers can simply transfer unowned stones.
Structure of an ownership block
			
{
   "ref_id":  "jh88hg3sxn43isjhdg0jsdjie03cu3",
   "type":    "sole",
   "sole": {
      "address_id": "jf8asdjji904fcn09jtunc909du3nec0envu9rg9"
   }
}
ref_id (string): This is the id of the stone (meta.id) where the ownership block is included. If the value is not the equal to meta.id, the block is considered invalid.

type (string): This is the type of ownership. Only sole ownership is currently supported. Expected value should be sole.

sole (object): Optional. This object contains information about the address that owns the stone. This is required when ownership.type is set to sole.

sole.address_id (string): This is the address id associated with the stone instance. Only signatures signed with the private key of this address will be accepted by the host.
Attributes Block
This is an optional block provided for issuers to include arbitrary data into the stone. Data included can be of any JSON data type. This block must be signed by the isser of the asset.
Structure of an ownership block
			
{
   "ref_id":  "jh88hg3sxn43isjhdg0jsdjie03cu3",
   "data": "data of any JSON type"
}
ref_id (string): This is the id of the stone (meta.id) where the ownership block is included. If the value is not the equal to meta.id, the block is considered invalid.

data (any): This field holds any issuer data. The value type can be string, number, JSON object or an array.

Embeds Block
To accomodate complex hierarchy of related assets, the embeds blocks allows multiple stones to be included in another stone.
Structure of an ownership block
			
{
   "ref_id":  "uahftaxnfajh88hg3sxn43isjhdg0jsdjie03cu3",
   "data": [
      {
         "meta": {
            "id": "h52kavcta6SVjdh738jfgsfijebsfhgjufkskdd",
            "type": "receipt",
            "created_at": 1458623859
         }
      },
      {
         "meta": {
            "id": "h52kavcta6SVjdh738jfgsfijebsfhgjufksjjj",
            "type": "bitcoin",
            "created_at": 1458623859
         },
         "ownership": {
            "ref_id": "h52kavcta6SVjdh738jfgsfijebsfhgjufksjjj",
            "type": "sole",
            "sole": {
               "address_id": "774hhgd888yby7rby7b88t900st93jsa"
            }
         }
      }
   ]
}
ref_id (string): This is the id of the stone (meta.id) where the ownership block is included. If the value is not the equal to meta.id, the block is considered invalid.

data (array): This must contain valid stone objects only.

Signatures Block
The signatures block contains the JWS signatures of all the other blocks. The base64 encoded string of this block is what gets transferred between users and applications. The benefit of using JWS for signing is the ability to rebuild a stone's JSON object from the signature. A stone will be considered invalid if a block's signature could not be verified.
Structure of a signatures block
			
{
   "meta": "eyk829xnd...hga0xa3aop1wsk",
   "ownership": "eqhauukfnh...ahhej23ndaa93smad",
   "attributes": "ey28ehdba9fhj...929uskhf8",
   "embeds": "8dhhdy289duh...2wjxjnjeks"
}
